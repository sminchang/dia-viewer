# dia-viewer

DiagramManifest JSON 파일(ERD · C4 아키텍처 · 플로우차트)을 인터랙티브하게 렌더링하는 정적 웹 뷰어입니다. 백엔드 없이 브라우저에서만 동작 — 매니페스트를 드래그·드롭으로 열면 자동 배치·라우팅되고, 노드를 직접 옮겨 다듬은 뒤 PNG/SVG · 단일 HTML 뷰어 · Layout JSON으로 내보냅니다.

`create-diagram` 같은 매니페스트 생성 스킬이 만든 결과물을 사람이 보고 다듬는 용도로 설계됐습니다.

## 사전 요구사항

- Node.js 20 (CI 기준 — 18 이상에서 동작)

## 빠른 시작

### 1. 클론 & 설치

```bash
git clone git@github.com:sminchang/dia-viewer.git
cd dia-viewer
npm install
```

### 2. 실행

```bash
npm run dev
```

브라우저에서 http://localhost:5175/dia-viewer/ 접속 후 매니페스트 JSON을 캔버스에 드래그·드롭하면 렌더링됩니다.

## 배포

`main` 브랜치에 푸시하면 GitHub Actions(`.github/workflows/deploy.yml`)가 GitHub Pages로 자동 배포합니다.

다른 곳에 올리려면:

```bash
npm run build
```

생성된 `dist/`를 Vercel · Netlify · S3 등에 정적 호스팅하면 됩니다. 단, 빌드 자산 경로가 GitHub Pages 서브패스(`/dia-viewer/`) 기준이므로 루트 도메인에 올릴 땐 `vite.config.ts`의 `base`를 조정하세요.

> 다운로드 메뉴의 **HTML (viewer)** 는 현재 화면을 그대로 담은 단일 HTML 파일을 내보냅니다. 인터넷 없이 더블클릭만으로 열려 하이라이트·줌·드래그·ERD 펼치기가 동작합니다. `npm run build`가 이 뷰어 템플릿(`build:viewer`)을 먼저 빌드해 메인 빌드에 포함시킵니다.

## 자동 배치 알고리즘

배치는 직접 옮기기 전의 **시작점**일 뿐이지만, 좋은 시작점이 손질 비용을 줄입니다. 다이어그램 종류에 따라 세 가지 파이프라인을 씁니다. (가중치·상수는 실제 매니페스트로 스윕·측정해 정한 값입니다 — `scripts/metrics.mts`)

### ERD — 계층형 배치 (ELK)

- **ELK `layered`** 알고리즘으로 FK 의존 관계를 계층(왼→오)으로 펼칩니다. 직교(`ORTHOGONAL`) 엣지 라우팅 + `LAYER_SWEEP` 교차 최소화, 노드 간격 60 / 계층 간격 120.
- 관계선은 **컬럼 단위**로 연결됩니다. 각 컬럼이 좌/우 핸들을 갖고, 두 테이블의 x 위치를 비교해 자연스러운 쪽을 동적으로 선택합니다(`updateErdHandles`).
- **같은 부모 컬럼**(타깃 테이블 + 컬럼)을 참조하는 FK는 같은 색을 공유합니다(6색 팔레트 순환) — "여기를 참조한다"가 색으로 읽힙니다.
- 카디널리티는 미드포인트 텍스트 대신 **까마귀발(crow's-foot, IE 표기)** 끝 마커로 표현합니다 — 위치에 독립적이고 ERD 산출물의 사실상 표준입니다.

### 아키텍처 — 교차 최소화 배치 (compact)

선 교차(crossing)를 최우선으로 줄이는 자체 파이프라인입니다. 서브도메인 클러스터링 없이 **공간 자체가 관계를 표현**하도록 둡니다.

1. **노드 분류** — person/external은 경계 밖 위성(satellite)으로 핀 고정(자기 피어에 가장 가까운 면에), 내부 컴포넌트만 슬롯 그리드에 배치.
2. **슬롯 그리드 시드** — 연결 수(degree) 내림차순으로 배치. 허브가 중앙을 먼저 차지하고, 이후 노드는 부분 그래프에 가장 잘 맞는 빈 슬롯에 그리디 배치.
3. **지역 탐색(local search)** — 이동(relocate)·교환(swap) 중 목적함수를 측정 가능하게 개선하는 변경만 채택, 한 패스에서 변화가 없으면 종료.
4. **목적함수(우선순위 순)** — 직선 교차 ≫ 노드 관통(pierce, 노드가 다른 선의 직선 경로 위에 있음) ≫ 종횡비 ≫ 엣지 길이 ≫ 캔버스 면적 ≫ 위로 흐르는 엣지(아래로 흐름 선호).
5. **멀티스타트(20회)** — 그리디 시드는 입력 순서에 민감해 한 번만 돌리면 "입력 순서 운"에 좌우됩니다. manifest 순서 + 결정적 셔플을 여러 번 시도하고, manifest 순서를 먼저 돌려 동점이 절대 퇴보하지 않게 합니다.
6. **라우터 인 더 루프(router-in-the-loop)** — 슬롯 목적함수는 내부 노드 사이 직선만 봅니다(위성·포트·트렁크는 안 보임). 그래서 각 후보를 실제 좌표로 펼쳐 **실제 라우터로 라우팅**한 뒤 시각 교차/길이를 측정해 최종 선택합니다.
7. **클린업 패스** — 직선 프록시가 잘못 떼어낸(stranded) 노드를 실제 라우터로 재판정해 이웃과 같은 행·열로 정렬합니다. 교차가 늘지 않고 경로가 짧아질 때만 채택(라우팅 예산으로 시간 제한). "선 따라 정렬되게 끌어다 놓기"를 자동화한 단계입니다.
8. **가로/세로 토글** — 최적 배치를 90° 회전해 반대 방향을 만듭니다. 회전은 **교차 불변**이라 두 방향이 같은 최소 교차를 공유하고, 뒤집어도 배치가 알아볼 수 있게 유지됩니다(A4 가로/세로 맞춤용). 정사각형에 가까우면 토글을 비활성화합니다.

### 아키텍처 엣지 라우팅 — 직교 장애물 회피 (Manhattan A*)

- 배치가 보장한 행/열 간격(gutter)을 통로 삼아, 모든 엣지를 **노드 박스를 통과하지 않는 직교 경로**로 그립니다.
- 희소 좌표 그리드 위 **A\*** 탐색 + 90° 꺾임당 비용 패널티(직선 선호).
- 포트(연결 면)는 규칙이 아니라 **측정된 경로 비용**으로 선택합니다 — 대각 쌍은 네 가지 면 조합(L자 포함)을 모두 라우팅해 가장 싼 실제 경로를 채택.
- 같은 출발 노드의 1:N 팬은 **공유 트렁크**로 묶고, 출발이 다른 선끼리 같은 통로를 지나면 **평행 레일**로 빗겨 분산합니다.
- 양방향 쌍은 양끝 화살표를 가진 **한 선으로 병합**(허브에서 출발). 단방향은 슬레이트, 양방향은 앰버로 색을 구분합니다.

### 플로우차트 — phase 선반 패킹 배치

흐름도는 step을 **phase(사용자가 경험하는 단계)** 영역으로 묶어 배치합니다. 아키텍처처럼 전역 교차를 최소화하기보다 **문서에 첨부 가능한 컴팩트함**을 우선합니다.

1. **phase 내부 서브배치** — 각 phase의 step을 위상정렬로 펼칩니다. 분기는 서브트리 폭에 비례해 벌리고, 합류는 부모들의 평균 열에 중앙 정렬하며, 윤곽(contour) 패킹으로 형제 서브트리를 실제 점유 칸만큼만 밀착시킵니다(얕은 형제가 깊은 팬 옆에 바짝 붙음).
2. **선반 패킹(shelf packing)** — phase 영역을 rank 순서로 주축(가로 방향이면 행, 세로면 열)을 따라 채우고 목표 폭을 넘으면 줄바꿈합니다. 작은 phase 둘이 나란히 들어가 긴 phase가 남긴 여백을 메웁니다.
3. **고아 방지** — 노드 약 2.2개보다 작은 phase는 자기 선반을 차지하지 않고 현재 선반에 overflow합니다. 작은 끝 phase가 새 선반 맨 앞(=앞 phase보다 시각적으로 먼저)에 떨어져 엣지가 거꾸로 흐르는 것을 막습니다.
4. **compact vs chain 측정 선택** — 여백을 채우는 패킹과 한 줄로 늘어놓는 체인(목표 폭=무한대) 두 후보를 만들어, **길이 + 꺾임 + 둘레**(교차 제외 — compact 우선)가 낮은 쪽을 채택합니다. 비슷한 크기 phase가 여럿이면 채우고, 한 phase가 지배적이면 체인이 이깁니다.
5. **방향 일관성** — 모든 phase의 진행축은 캔버스 방향(가로→오른쪽, 세로→아래)을 따릅니다. 구조가 같은 phase(선형 체인 둘)가 같은 방향으로 읽히고, 분기 phase도 같은 방향으로 흐르되 fan만 옆으로 퍼집니다.
6. **터미네이터·경로 부호화** — 시작점(in-degree 0)·끝점(out-degree 0)은 **알약(pill)** 모양으로 표시합니다. `data.path`별로 선 색을 나눠(실패류 경로는 rose) 한 경로를 색으로 따라 읽을 수 있습니다.
7. **가로/세로 토글** — 아키텍처처럼 회전하지 않고, 각 방향마다 선반 주축을 바꿔 **따로 패킹**합니다(텍스트 노드의 가로로 긴 종횡비 탓에 90° 회전은 간격이 깨지기 때문). 가로는 넓게, 세로는 높게 — 문서 지면에 맞춰 고릅니다.

### 플로우차트 엣지 라우팅 — 아키텍처에서 포크한 Manhattan A*

기반 동작은 아키텍처 라우터(직교 A*, 트렁크 번들, 평행 레일 분산, 양방향 amber 병합)와 같지만, 흐름도는 요구가 달라질 수 있어 **독립 파일로 포크**해 따로 관리합니다. 차이점:

- **대칭 분기 팬(comb)** — 한 노드에서 갈라지는 1:N 단방향 분기는 공유 줄기에서 나와 **하나의 split line에서 꺾여** 각 타깃으로 떨어집니다. 아키텍처는 가지마다 가장 싼 곳에서 꺾지만, 흐름도의 결정점은 대칭으로 읽혀야 하기 때문입니다(통로가 막힌 가지만 A*로 폴백).
- **교차·꺾임 우선 combo 선택** — 출구·도착 면 16가지 조합을 모두 실제로 라우팅해 측정 비용으로 고릅니다(자연스러운 4개를 앞에 둬 동점은 흐름 방향 유지). 교차는 꺾임보다(CROSS_PENALTY > BEND_WORTH), 꺾임은 길이보다 비싸게 매겨, "짧지만 교차/꺾임이 많은" 경로가 이기지 못하게 합니다.
- **골격 먼저, 백엣지 나중(skeleton-then-fit)** — 짧은 엣지를 먼저 라우팅해 골격을 세우고, 길게 되돌아가는 엣지를 마지막에 라우팅합니다. 그래야 백엣지가 완성된 골격을 보고 자신이 만들 교차를 세서, 둘레로 돌아 피할 수 있습니다(그리디 라우팅의 순서 의존성 보완).
- **phase 경계 클리어런스** — phase 헤더(어두운 띠)를 가로지르는 수평선은 헤더 위로 올리고, 박스 벽에 올라타는 둘레 엣지는 벽 바깥으로 밀어 경계선과 간격을 둡니다. 라우팅 후처리이며, 서로 다른 엣지를 떼어놓는 레인 분산이 최종 결정권을 갖도록 그 앞에서 실행합니다.
- **도착측 라벨 배치** — 조건 라벨을 도착 포트에서 고정 거리만큼 앞에 앵커하고, 그 자리가 모호하면(다른 라벨·남의 선이 박스에 들어오면) **자기 선을 따라서만 슬라이드**해 어느 선의 라벨인지 귀속을 보장합니다. 흐름도는 주석을 켠 채 보고서에 넣는 용도라 라벨이 문서급(12px)이고 기본 ON입니다.
